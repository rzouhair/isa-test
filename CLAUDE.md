# Poke - AI Context Guide

> This file provides context for AI assistants (Claude Code, OpenCode, Cursor, etc.)

## Project Overview

Poke is an iOS app built with **SwiftUI** and **Swift 6**, following **MVVM + Clean Architecture**. Subscriptions are managed through **RevenueCat**.

## Architecture

```
poke/
│
├── Application/
│   App lifecycle and bootstrap. AppMain.swift is the @main entry point
│   that wires up SwiftData, RevenueCat, AI services, and environment
│   injection. AppState holds global state. RootView is the top-level
│   container that decides what to show (onboarding vs main app).
│
├── Core/
│   ├── DI/
│   │   Registers and resolves dependencies. Add new service bindings
│   │   here when introducing a new protocol/implementation pair.
│   │
│   └── Navigation/
│       Router.swift defines a Route enum for every navigable destination.
│       Add new cases here when creating new screens. Uses NavigationStack.
│
├── Data/
│   ├── Datasources/
│   │   Low-level data access. SwiftDataDatasource wraps ModelContext
│   │   queries. UserDefaultsDatasource stores simple key-value prefs.
│   │   New persistence backends go here.
│   │
│   ├── Repositories/
│   │   Mediates between datasources and the rest of the app. Conforms
│   │   to protocols defined in Domain/Protocols. Business logic should
│   │   call repositories, never datasources directly.
│   │
│   └── Services/
│       External API integrations (network calls). Each service handles
│       a single external dependency. Add new API clients here.
│
├── Domain/
│   ├── Models/
│   │   Data structures shared across the app. SwiftData @Model classes
│   │   live here. Keep models free of UI or networking logic.
│   │
│   ├── Protocols/
│   │   Interfaces that decouple layers. Repositories and services
│   │   conform to these. Define new protocols here before implementing.
│   │
│   ├── Enums/
│   │   Shared enumerations used across multiple features.
│   │
│   ├── Functions/
│   │   AI function-calling definitions (tool use). Each file defines
│   │   a function the AI model can invoke. Register new functions in
│   │   AppMain.swift's functionsToRegister array.
│   │
│   ├── Prompts/
│   │   System/user prompt templates sent to the AI backend.
│   │   Keep prompt text here, not inline in ViewModels.
│   │
│   └── Utilities/
│       Parsing helpers and shared utilities with no UI dependency.
│
├── Features/
│   Each subfolder is a self-contained feature module. A feature
│   typically has a View (SwiftUI) and a ViewModel (@Observable).
│   Features should not import each other directly — communicate
│   through Router navigation or shared Domain models.
│   │
│   ├── Home/              Main dashboard / landing screen.
│   ├── CameraCapture/     Camera interface, photo picker, crop frame.
│   ├── Settings/          App settings, mail composer, about page.
│   ├── Purchases/         Paywall and subscription management.
│   │   ├── Data/          SubscriptionsRepository for purchase state.
│   │   ├── Utils/         StoreKit/RevenueCat helper extensions.
│   │   └── Views/         Paywall screen variants (Yearly, Reviews).
│   ├── OnboardingFlow/    Multi-step onboarding with permission requests.
│   └── .../               Add new feature folders here as needed.
│
├── Presentation/
│   ├── Components/
│   │   Reusable, feature-agnostic UI building blocks (tab bar, cards,
│   │   badges, buttons, etc.). If a view is used in 2+ features, it
│   │   belongs here.
│   │
│   ├── Extensions/
│   │   Swift and SwiftUI type extensions (Font, UIImage, String, etc.).
│   │   Keep extensions small and focused on a single type.
│   │
│   ├── Resources/
│   │   Asset catalogs: Colors.xcassets, Images.xcassets, Fonts/.
│   │   Add new colors/images here and run SwiftGen to regenerate.
│   │
│   └── Generated/
│       Auto-generated by SwiftGen. Do NOT edit manually — changes
│       are overwritten. Regenerate with `swiftgen` after asset changes.
│
├── Constants.swift
│   App-wide configuration: API keys, URLs, app metadata, product IDs.
│   Update this when changing environments or branding.
│
└── Item.swift
    General-purpose data model.
```

## App Entry Point

`AppMain.swift` bootstraps:
- SwiftData `ModelContainer`
- RevenueCat configuration
- AI service + function registration
- Environment injection (`AppState`, `AIService`, `SwiftDataDatasource`)

## Code Conventions

- **Swift 6** with strict concurrency
- **@Observable** macro for ViewModels (not ObservableObject)
- **@State, @Binding, @Bindable** for SwiftUI state
- **async/await** for all async work (no Combine for new code)
- **Value types** (structs) preferred over classes, except SwiftData @Model
- **Protocol-oriented** design with protocol extensions for shared behavior
- camelCase for variables/functions, PascalCase for types
- Boolean properties: `is`/`has`/`should` prefixes
- Methods named with verbs: `fetchData()`, `loadItems()`

## Navigation

`Router.swift` uses a typed `Route` enum with `NavigationStack`. Tab bar is a custom `HighlightTabBar` with a floating center camera button.

## Key Dependencies

- **RevenueCat** - Subscription management
- **SwiftData** - Local persistence
- **AVFoundation** - Camera capture
- **SwiftGen** - Asset code generation
- **Inject** - Hot reload during development
- **PostHog** - Analytics
- **Sentry** - Error & crash tracking

## Guidelines

- Follow Apple Human Interface Guidelines
- Support dark mode and dynamic type
- Use SF Symbols for icons
- Handle all screen sizes (iPhone focus)
- Lazy load views and images for performance
- Each feature module should be self-contained with its own View + ViewModel
- Keep ViewModels testable - inject dependencies via protocols
- Use `Result` type or throwing functions for error handling
